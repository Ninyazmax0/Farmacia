OBJETIVOS DEL PROYECTO
Objetivo General

Desarrollar e implementar un sistema digital de inventario para la Farmacia Santa Cruz que permita organizar y controlar los medicamentos, reduciendo las pérdidas económicas y mejorando la eficiencia operativa del negocio.

Objetivos Específicos
  • Registrar todos los medicamentos y productos de uso personal en el sistema digital.
  • Controlar el stock de productos mediante alertas de bajo inventario.
  • Detectar productos próximos a vencer para evitar pérdidas económicas.
  • Facilitar la búsqueda de medicamentos por múltiples criterios (nombre, código, categoría).
  • Generar reportes de inventario, ventas, compras y vencimientos para toma de decisiones.
  • Registrar compras a proveedores con actualización automática de stock y precios.
  • Registrar ventas de productos con descuento automático de stock.
  • Monitorear cambios en precios de costo para ajustar precios de venta.
ALCANCE DEL SISTEMA
Alcance Funcional

El sistema de inventario para la Farmacia Santa Cruz incluye las siguientes funcionalidades:

  • Autenticación: Inicio de sesión para administradores y vendedores con control de acceso por roles.
  • Gestión de Productos: Registro, edición y eliminación de productos con información completa (nombre, precio, stock, laboratorio, fecha de vencimiento, categoría).
  • Gestión de Compras: Registro de compras a proveedores con actualización automática de stock y precios de costo.
  • Gestión de Ventas: Registro de ventas con cálculo automático de totales, generación de facturas y descuento de stock.
  • Monitoreo de Costos: Detección de cambios en precios de costo y sugerencia de ajustes en precios de venta.
  • Alertas de Caducidad: Generación automática de alertas para productos próximos a vencer (30 días).
  • Reportes: Generación de reportes de inventario, ventas, compras y vencimientos con exportación a PDF.
Alcance No Funcional
  • Usuarios del Sistema: Solo administradores y vendedores de la farmacia. NO se registran clientes que compran productos.
  • Alcance Geográfico: Local físico de la Farmacia Santa Cruz en San Martín Norte.
  • Tipo de Sistema: Sistema de escritorio/local, NO es una página web ni sistema en línea.
  • Base de Datos: Local, almacenada en el mismo equipo donde se ejecuta el sistema.

DIAGRAMA DE ESTADOS
Diagrama de Estados – Ciclo de Vida de un Producto

Estados de un Producto:

  • Registrado: Producto ingresado al sistema con todos sus datos.
  • Activo: Producto disponible para la venta.
  • Stock Bajo: Producto con cantidad menor al stock mínimo establecido.
  • Próximo a Vencer: Producto con fecha de vencimiento en los próximos 30 días.
  • Vencido: Producto que ha superado su fecha de vencimiento.
  • Agotado: Producto sin stock disponible.
  • Inactivo: Producto deshabilitado por el administrador.

Transiciones:

  • Registrado → Activo: Cuando se guarda por primera vez.
  • Activo → Stock Bajo: Cuando el stock baja del mínimo.
  • Activo → Próximo a Vencer: Cuando faltan 30 días para vencer.
  • Próximo a Vencer → Vencido: Cuando se supera la fecha de vencimiento.
  • Activo → Agotado: Cuando el stock llega a 0.
  • Agotado → Activo: Cuando se registra una compra que repone stock.
  • Activo → Inactivo: Cuando el administrador lo deshabilita.
  • Inactivo → Activo: Cuando el administrador lo rehabilita.

PROBLEMÁTICA DEL PROYECTO

La Farmacia Santa Cruz, ubicada en el pasaje parroquial y avenida norte, San Martín Norte, gestiona anualmente $12,000 en medicamentos y productos de uso personal. Sin embargo, su modelo de registro manual le ha causado distintos conflictos:

  • Disminución de clientes por falta de organización en el inventario.
  • Pérdida económica de $400 en los últimos 2 años debido a productos vencidos no detectados a tiempo.
  • Dificultad para conocer existencias reales de medicamentos.
  • Problemas para detectar cambios de precios en compras a proveedores.
  • Ineficiencia en la búsqueda de productos para empleados.
  • Falta de reportes que permitan tomar decisiones basadas en datos.

La digitalización del inventario no es un capricho, es una necesidad preventiva para asegurar una mayor organización para los empleados y el propio dueño.

SOLUCIÓN PROPUESTA

Desarrollar un prototipo de sistema digital para la gestión de inventario que permita:

  • Controlar entrada y salida de medicamentos y productos médicos.
  • Detectar cambios de precios y movimientos en tiempo real.
  • Generar alertas de productos próximos a vencer.
  • Facilitar búsqueda de medicamentos por múltiples criterios.
  • Generar reportes de compras, facturas y vencimientos.

Con este sistema se ha detectado que se puede lograr:

  • 40% de reducción de pérdidas
  • 30% de ahorro por registros exactos

CASO DE USO CU-04: Registrar Ventas
CódigoCU-04
NombreRegistrar Ventas
Actor PrincipalVendedor
Precondiciones1. Usuario autenticado.
2. Productos existentes en inventario con stock disponible.
Flujo Principal1. Vendedor selecciona productos a vender.
2. Sistema calcula subtotal.
3. Vendedor ingresa cantidad.
4. Sistema calcula total.
5. Vendedor confirma la venta.
6. Sistema genera factura.
7. Sistema descuenta stock del inventario.
Flujos AlternativosFA-01: Producto sin stock → advertencia al vendedor.
Postcondiciones1. Venta registrada.
2. Factura generada.
3. Stock actualizado.

CASO DE USO CU-05: Monitoreo de Costos
CódigoCU-05
NombreMonitoreo de Costos
Actor PrincipalAdministrador
Precondiciones1. Administrador autenticado.
2. Registro de compras disponible en BD.
Flujo Principal1. Administrador selecciona producto.
2. Sistema muestra historial de precios de costo.
3. Sistema detecta cambios de precio.
4. Sistema sugiere ajuste de precio de venta.
5. Administrador confirma o rechaza el ajuste.
Flujos AlternativosFA-01: Sin cambios detectados → no generar sugerencia.
Postcondiciones1. Precios actualizados si el admin lo confirma.

CASO DE USO CU-06: Alerta de Caducidad
CódigoCU-06
NombreAlerta de Caducidad
Actor PrincipalAdministrador
Precondiciones1. Productos con fecha de vencimiento registrada en BD.
Flujo Principal1. Sistema verifica fechas de vencimiento diariamente.
2. Sistema identifica productos próximos a vencer (30 días).
3. Sistema genera alerta visible.
4. Sistema notifica al Administrador.
Flujos AlternativosFA-01: Sin productos próximos a vencer → no generar alerta.
Postcondiciones1. Alertas visibles en el panel del sistema.

CASO DE USO CU-07: Generar Reportes
CódigoCU-07
NombreGenerar Reportes
Actor PrincipalAdministrador
Precondiciones1. Administrador autenticado.
2. Datos disponibles en el sistema.
Flujo Principal1. Administrador selecciona tipo de reporte (inventario, ventas, compras, vencimientos).
2. Sistema consulta los datos necesarios.
3. Sistema genera el reporte.
4. Sistema muestra vista previa.
5. Administrador exporta o imprime.
Flujos AlternativosFA-01: Sin datos disponibles → mostrar mensaje de advertencia.
Postcondiciones1. Reporte generado correctamente.
